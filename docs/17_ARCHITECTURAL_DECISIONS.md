ADR-001: استخدام Supabase كمنصة خلفية (BaaS)
السياق: المشروع بدأ كمشروع فردي. بناء خادم خلفي كامل (Node.js/Express أو Django) كان سيستغرق شهوراً إضافية. كان المطلوب حل سريع وموثوق يوفر: قاعدة بيانات، مصادقة، استضافة دوال خادم، وتخزين ملفات.
القرار: اخترنا Supabase (مفتوح المصدر) كمنصة Backend-as-a-Service.
البدائل المرفوضة:
البديل	لماذا رُفض؟
Firebase	قاعدة بيانات NoSQL (Firestore) غير مناسبة للعلاقات المعقدة في ERP
خادم Node.js/Express مخصص	يحتاج إدارة خادم، أمان، وتوسع يدوي. تكلفة تشغيلية عالية
AWS RDS + Lambda	تعقيد إضافي في الإعداد. Supabase يقدم حلاً متكاملاً أبسط
الأسباب الموجبة:
•	PostgreSQL كاملة مع RLS (أمان على مستوى الصفوف) – أساسي لـ Multi-Tenant
•	Auth جاهز (سجل دخول، جلسات، JWT)
•	Edge Functions (Deno/TypeScript) – قريبة من قاعدة البيانات، مما يقلل زمن الاستجابة
•	تكلفة منخفضة: خطة مجانية سخية، وخطط مدفوعة تتناسب مع النمو
•	مفتوح المصدر (يمكن الترحيل منه مستقبلاً إذا لزم)
العواقب:
•	✅ إيجابيات: تم بناء النظام بالكامل في 5 أشهر بدلاً من سنة. PostgreSQL علائقية قوية. RLS مدمج. لا نحتاج DevOps.
•	⚠️ سلبيات: Vendor Lock-in (اعتماد على مزود خارجي). Edge Functions لها قيود (Timeout، ذاكرة).
المقارنة بالمنافسين:
•	SAP B1: يعتمد على MS SQL Server. ترخيصه مكلف.
•	Odoo: يعتمد على PostgreSQL أيضاً، لكنه يحتاج خادم تطبيقات.
•	الروائع يستغني عن الخادم تماماً.

ADR-002: فرض Vanilla JavaScript ES5 على تطبيقات PWA
السياق: السوق العربي يحتوي على أجهزة قديمة (Android 6، ذاكرة 2GB، متصفحات قديمة). عمال المخازن والسائقون يستخدمون هذه الأجهزة. أي استخدام لـ ES6+ يؤدي إلى انهيار التطبيق على أجهزتهم.
القرار: استخدام Vanilla JavaScript ES5 (var و function فقط) لجميع تطبيقات PWA والنظام الأم.
البدائل المرفوضة:
البديل	لماذا رُفض؟
React / Vue / Angular	أطر عمل ثقيلة، تزيد حجم التطبيق، وتحتاج خطوة بناء (Build step)
TypeScript	يضيف طبقة تعقيد ويحتاج Transpiler
ES6+ بشكل كامل	بعض الأجهزة القديمة جداً لا تدعمه بالكامل
استخدام Babel/SWC	يتطلب خطوة بناء معقدة. النظام بُني ليكون بسيطاً (ملفات HTML مباشرة)
الأسباب الموجبة:
•	توافق أقصى: ضمان تشغيل التطبيق على أي متصفح حديث أو قديم نسبياً دون أخطاء
•	لا خطوة بناء: التطبيق عبارة عن ملفات HTML/JS تُنشر مباشرة
•	حماية من الأخطاء: const و let قد يسببان أخطاء SyntaxError في متصفحات قديمة
العواقب:
•	✅ إيجابيات: توافق مع أي متصفح. لا Build Step. سريع التحميل. خفيف على الأجهزة الضعيفة.
•	⚠️ سلبيات: إنتاجية أقل من ES6+. كود أطول. عدم وجود Type Safety.
تحديث (دستور 2.0): تم تخفيف القيد للسماح بـ const للثوابت العامة و let في الحلقات، مع إمكانية استخدام Babel للتوافق. لكن var و function يظلان الأساس.
المقارنة بالمنافسين:
•	SAP B1: يستخدم تقنيات خاصة (SAP UI5) ثقيلة.
•	Odoo: يستخدم JavaScript حديث مع Owl framework.
•	Daftra: يستخدم React (SPA).
•	الروائع هو الوحيد الذي يفرض ES5 للتوافق مع الأجهزة القديمة جداً – ميزة تنافسية حقيقية.

ADR-003: اعتماد PWA + Offline-First بدلاً من تطبيق Native
السياق: السائقون والمندوبون يعملون في مناطق نائية بدون إنترنت. احتجنا تطبيقات تعمل Offline وتتزامن عند عودة الاتصال. وفي نفس الوقت، لا نريد تطوير تطبيقين (iOS + Android).
القرار: اخترنا PWA (Progressive Web Application) مع Offline-First عبر Dexie.js (IndexedDB) و Service Worker.
البدائل المرفوضة:
البديل	لماذا رُفض؟
React Native / Flutter	تكلفة تطوير مضاعفة (iOS + Android). لا حاجة للـ Native APIs حالياً
تطبيق ويب فقط (بدون Offline)	لا يعمل بدون إنترنت. يفشل في السوق المستهدف
الأسباب الموجبة:
•	قاعدة كود واحدة (HTML/JS/CSS) تعمل على جميع المنصات
•	لا حاجة لمتجر تطبيقات (Google Play/App Store) – التثبيت مباشر من المتصفح
•	Service Worker يوفر Offline وإشعارات
•	التكلفة أقل بكثير (لا حاجة لمطورين متخصصين في Android/iOS)
العواقب:
•	✅ إيجابيات: وصول أسرع للسوق. صيانة أسهل (كود واحد). Offline-First حقيقي.
•	⚠️ سلبيات: لا يمكن الوصول الكامل لكاميرا/GPS/Bluetooth. بعض القيود في iOS.
المقارنة بالمنافسين:
•	SAP: تطبيق Native (SAP Fiori).
•	Odoo: PWA + تطبيق Native.
•	الروائع اختار PWA للسرعة والتكلفة. (P3: تطبيق Native).

ADR-004: استخدام Dexie.js (IndexedDB) لتحقيق Offline-First
السياق: مندوبو المبيعات وسائقو التوصيل يعملون في مناطق نائية. التطبيق يجب أن يستمر في العمل بدون اتصال، ثم المزامنة عند عودة الاتصال.
القرار: استخدام مكتبة Dexie.js (غلاف لـ IndexedDB) لتخزين البيانات محلياً، مع طابور pending_updates للعمليات المؤجلة.
البدائل المرفوضة:
البديل	لماذا رُفض؟
LocalStorage	سعة محدودة (5MB)، وبطيء مع كميات البيانات الكبيرة
IndexedDB الأصلية	API معقد جداً
PouchDB/CouchDB	مزامنة قوية لكنها معقدة وتتطلب خادم CouchDB إضافي
الأسباب الموجبة:
•	سعة كبيرة: IndexedDB غير محدودة عملياً
•	أداء: استعلامات سريعة عبر Dexie
•	مزامنة بسيطة: pending_updates يضمن عدم فقدان أي عملية
•	core.js (وحدة RW_DB) توحد استخدامه عبر جميع التطبيقات
العواقب:
•	✅ إيجابيات: تطبيقات المبيعات تعمل Offline بالكامل. تخزين غير محدود نسبياً.
•	⚠️ سلبيات: بعض التطبيقات لم تهاجر إلى core.js بعد وتستخدم Dexie.js مباشرة (ازدواجية).
المقارنة بالمنافسين:
•	معظم منافسينا (Odoo, Daftra) لا يقدمون Offline-First حقيقياً. هذا يُعد ميزة تنافسية قوية للروائع.

ADR-005: نمط "الأصل والتجميع" – order_details كمصدر وحيد للحقيقة
السياق: order_details يسجل الكميات لكل أوردر. run_sheet_details يجمع الكميات لكل رانشيت. إذا كان هناك مصدران للحقيقة، سيحدث تضارب. أردنا مصدراً واحداً.
القرار: order_details هو الأصل والمصدر الوحيد للحقيقة. run_sheet_details هو مجرد إسقاط (Projection) يُعاد بناؤه من order_details عبر sync-run-sheet-details ولا يُلمس مباشرة.
البدائل المرفوضة:
البديل	لماذا رُفض؟
تحديث run_sheet_details مباشرة	سيؤدي إلى تناقض البيانات مع order_details عند حدوث أي خطأ
دمج الجدولين في جدول واحد	لا يمكن تتبع التاريخ. لا يمكن إعادة البناء
الأسباب الموجبة:
•	هذا هو نمط CQRS (Command Query Responsibility Segregation) مصغر: order_details هو الـ Write Model، و run_sheet_details هو الـ Read Model
•	أي تغيير في الكميات يحدث في order_details فقط
•	sync-run-sheet-details تعيد بناء run_sheet_details من order_details عند الحاجة
•	حماية من العبث: لا يمكن لموظف تعديل run_sheet_details مباشرة
العواقب:
•	✅ إيجابيات: لا تضارب بين المصدرين. سجل التدقيق واضح. إمكانية إعادة البناء.
•	⚠️ سلبيات: إذا فشل sync-run-sheet-details، run_sheet_details تصبح قديمة.
المقارنة بالمنافسين:
•	هذا النمط غير موجود في الأنظمة المنافسة. هم يحدثون جداول التجميع مباشرة، مما قد يؤدي لفقدان البيانات.
•	SAP: يستخدم Materialized Views مع تحديث دوري.
•	Odoo: يستخدم recompute() عند كل تغيير.

ADR-006: تصميم آلية الـ 6 كميات في order_details
السياق: في أنظمة التوزيع، البضاعة تمر بعدة مراحل. معظم الأنظمة تتعامل مع هذا عبر جداول منفصلة (SAP: Delivery + Goods Issue + Return). أردنا حلاً أبسط.
القرار: 6 أعمدة في صف واحد في order_details: qty_ordered, qty_picked, qty_loaded, qty_delivered, qty_refused, qty_returned.
البدائل المرفوضة:
البديل	لماذا رُفض؟
جداول منفصلة لكل مرحلة (مثل SAP)	معقد. يحتاج ربط جداول متعددة لكل مرحلة
جدول واحد مع عمود status فقط	لا يوفر تتبع الكميات
الأسباب الموجبة:
•	بساطة الاستعلام: تقرير واحد يجيب عن "أين ذهبت البضاعة؟"
•	تقليل التعقيد: لا حاجة لربط جداول متعددة لكل مرحلة
•	الشفافية: أي فرق بين الكميات يظهر فوراً (العجز = loaded - delivered - returned)
•	هذا التصميم لا يوجد في أي نظام منافس. هو "سر الصنعة".
العواقب:
•	✅ إيجابيات: تتبع فائق الدقة للبضاعة. تقارير فورية عن الفاقد والعجز.
•	⚠️ سلبيات: صف واحد يحتوي على 6 أعمدة رقمية – قد يبدو "غير مطبع" (Denormalized). لكن هذا مقصود.

ADR-007: sync-run-sheet-details كطبقة وسطى للتجميع
السياق: run_sheet_details يجب أن تعكس مجموع الكميات من جميع order_details المرتبطة بالرانشيت. أردنا آلية مركزية تضمن تحديثها.
القرار: Edge Function تستدعى بعد كل complete-* (يدوياً حالياً)، مع خطة للتحويل إلى PostgreSQL Trigger (P2).
البدائل المرفوضة:
البديل	لماذا رُفض؟
View (مشهد) في PostgreSQL	سيء الأداء مع بيانات كبيرة
Trigger من البداية	كان سيعقد التطوير المبكر
الأسباب الموجبة:
•	Edge Function توفر مرونة أكبر في البداية (يمكن تعديل المنطق بسهولة)
•	Trigger سيكون أكثر كفاءة (لا حاجة لاستدعاء خارجي)
العواقب:
•	✅ إيجابيات: run_sheet_details محدثة دائماً (إذا تم استدعاء الدالة).
•	⚠️ سلبيات: إذا فشل الاستدعاء أو نُسي، تحدث كارثة صامتة. تم إصلاح استدعاء invoke واستبداله بـ fetch اليدوي.

ADR-008: استخراج core.js كنواة موحدة
السياق: بعد بناء 26 تطبيق PWA، اكتشفنا أن كل تطبيق يعيد تعريف: المصادقة، الاتصال بـ Supabase، Dexie.js، دوال UI، كاش الصور، Service Worker. هذا تسبب في تكرار هائل للكود (~40,000 سطر مكرر).
القرار: استخراج كل الكود المتكرر في ملف core.js واحد يحتوي على 6 وحدات: RW_Auth, RW_DB, RW_API, RW_UI, RW_ImageCache, RW_SW.
البدائل المرفوضة:
البديل	لماذا رُفض؟
الاستمرار في النسخ واللصق	صيانة مستحيلة. أي تغيير يتطلب تعديل كل التطبيقات
استخدام إطار عمل (React/Vue)	كان سيضيف تعقيداً غير ضروري لمشروع Vanilla JS
npm package	لا نستخدم Build Step
الأسباب الموجبة:
•	اختصر 40,000 سطر إلى 500 سطر نواة
•	أي تعديل في المصادقة أو API يُطبق مرة واحدة
•	يفرض التوحيد (Standardization) عبر جميع التطبيقات
العواقب:
•	✅ إيجابيات: صيانة أسهل بكثير. تقليل الأخطاء (منطق موحد).
•	⚠️ سلبيات: 8 تطبيقات فقط مهاجرة إلى core.js. 18 تطبيقاً ما زالت تستخدم الكود المكرر (فجوة P1).

ADR-009: حظر supabase.functions.invoke وفرض fetch اليدوي
السياق: اكتشفنا أن supabase.functions.invoke يخفي تفاصيل الطلب. عندما يفشل استدعاء Edge Function، لا نعرف لماذا. هذا تسبب في انهيارات صامتة.
القرار: استخدام fetch اليدوي مع Authorization: Bearer في جميع الحالات. حظر supabase.functions.invoke نهائياً (المادة 1 من الدستور). core.js (وحدة RW_API) تغلف fetch اليدوي.
البدائل المرفوضة:
البديل	لماذا رُفض؟
supabase.functions.invoke	تخفي التفاصيل، لا تسمح بتخصيص كامل للـ Headers، قد تفشل بسبب CORS دون رسالة خطأ
الأسباب الموجبة:
•	تحكم كامل: نستطيع قراءة rawBody وتسجيله لتصحيح الأخطاء
•	أمان: نضمن إرسال Authorization الصحيح يدوياً
•	توافق: fetch متاحة في جميع المتصفحات وبيئات Deno
العواقب:
•	✅ إيجابيات: تتبع أفضل للأخطاء. التزام بالدستور.
•	⚠️ سلبيات: بعض المواضع ما زالت تستخدم invoke (لم تُصلح بعد: complete-picking, RW_OnlineStore, RW_Purchases).

ADR-010: منع Hard Delete واستخدام Soft Delete وقيد عكسي
السياق: في الأنظمة المحاسبية (مثل SAP)، لا يمكن حذف أي حركة مالية. أي خطأ يُصحح بقيد عكسي. أردنا نفس المستوى من الدقة.
القرار:
•	لا Hard Delete للبيانات المالية (الأوردرات بعد التأكيد، الفواتير، القيود، حركات المخزون)
•	الأخطاء تُصحح بـ قيد عكسي (Reversal) أو تسوية
•	الحذف مسموح فقط للسجلات التشغيلية غير المرتبطة بسجلات مالية
البدائل المرفوضة:
البديل	لماذا رُفض؟
Hard Delete مع Audit Log	يفقد البيانات الأصلية. لا يمكن تتبع "من حذف ماذا"
Soft Delete لكل شيء	تراكم بيانات غير ضرورية
العواقب:
•	✅ إيجابيات: تتبع كامل. لا يمكن لأي موظف إخفاء خطأه. متوافق مع معايير التدقيق المالي.
•	⚠️ سلبيات: تعقيد في الكود. قاعدة البيانات تنمو بسرعة.

ADR-011: استخدام Edge Functions (Deno) بدلاً من خادم تقليدي
السياق: احتجنا إلى منطق خلفي (Business Logic) لتنفيذ العمليات. الخياران: خادم Node.js تقليدي، أو Serverless Functions.
القرار: اخترنا Supabase Edge Functions (Deno/TypeScript). كل عملية هي دالة مستقلة تُستدعى عبر HTTP.
البدائل المرفوضة:
