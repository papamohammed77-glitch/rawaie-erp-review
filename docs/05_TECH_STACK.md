05_TECH_STACK.md – توثيق متعمق لجميع التقنيات والاعتماديات
١. الغرض من هذه الوثيقة
توثيق شامل لكل تقنية ومكتبة وخدمة يعتمد عليها "الروائع ERP". هذه الوثيقة هي المرجع الرسمي لأي مهندس معماري أو مدير تقني لفهم:
•	مما صُنع النظام؟
•	لماذا تم اختيار كل تقنية؟
•	كيف ترتبط التقنيات ببعضها البعض؟
•	ما هي المخاطر والبدائل؟
________________________________________
٢. نظرة عامة على المعمارية التقنية
"الروائع ERP" مبني على معمارية Jamstack هجينة: واجهة أمامية ثابتة (HTML/CSS/JS) تُنشر عبر CDN، مع خدمات خلفية (BaaS) تُدار بواسطة Supabase. لا يوجد خادم وسيط (Middleware Server). كل المنطق الخلفي يعمل عبر Edge Functions (Serverless) قريبة من قاعدة البيانات.

٣.١ الواجهة الأمامية (Frontend – PWA)
3.1.1 HTML5 + CSS3 (Tailwind CSS)
•	ماذا يفعل؟ يوفر هيكل الصفحات وتنسيقها. Tailwind CSS هو إطار عمل CSS يُحمّل عبر CDN، يسمح بكتابة تنسيقات سريعة باستخدام Classes جاهزة.
•	لماذا اختير؟ إنتاجية عالية، تصميم متناسق، لا حاجة لملفات CSS منفصلة. متجاوب مع جميع أحجام الشاشات. متوافق مع المتصفحات القديمة.
•	كيف يُستخدم؟ <script src="https://cdn.tailwindcss.com"></script> في كل ملف HTML (26 تطبيق PWA + main.html).
•	الاعتماديات: CDN خارجي.
•	من يعتمد عليها: جميع تطبيقات PWA والنظام الأم.
•	البدائل المرفوضة: Bootstrap (أقل مرونة)، CSS مخصص (يبطئ التطوير).
•	المخاطر: الاعتماد على CDN. إذا تعطل، ينهار التصميم. الحل: تنزيل النسخة محلياً (P2).
3.1.2 Vanilla JavaScript ES5 صارم
•	ماذا يفعل؟ يوفر منطق التطبيق بالكامل (المصادقة، جلب البيانات، إدارة الحالة، التفاعل مع المستخدم).
•	لماذا اختير؟ هذا هو "سر قوة" النظام في السوق العربي. يضمن التوافق مع أجهزة Android 6 القديمة التي يستخدمها السائقون وعمال المخازن. لا يحتاج Transpiler أو Build Tools. الكود يُنشر كما هو. دستور الروائع 2.0 سمح بـ const و let بشروط، لكن var يبقى الأساس.
•	كيف يُستخدم؟ كل تطبيق PWA هو ملف HTML يحتوي على كود JavaScript في <script> أو عبر core.js. core.js (6 وحدات) هو النواة المشتركة التي توحد الكود المتكرر.
•	الاعتماديات: core.js (للتطبيقات الذهبية)، Supabase SDK، Dexie.js، SweetAlert2.
•	من يعتمد عليها: كل شيء في الواجهة الأمامية.
•	البدائل المرفوضة: React/Vue/Angular (تتطلب Transpiler، لا تدعم أجهزة قديمة)، TypeScript (يحتاج Compiler).
•	المخاطر: إنتاجية أقل قليلاً من ES6+. لكن الدستور 2.0 سمح بـ const للثوابت و let في الحلقات.
•	مقارنة بالمنافسين:
o	SAP: يستخدم SAPUI5 (JavaScript خاص).
o	Odoo: يستخدم JavaScript ES6+ مع Owl Framework.
o	دفترة: يستخدم React (ES6+).
o	الروائع هو الوحيد الذي يفرض ES5 للتوافق مع الأجهزة القديمة جداً.
3.1.3 SweetAlert2
•	ماذا يفعل؟ يوفر نوافذ منبثقة (مودالات) جميلة وتفاعلية: تأكيدات، تنبيهات، Toast، Loader، نوافذ إدخال بيانات معقدة.
•	لماذا اختير؟ تجربة مستخدم أفضل من alert() و confirm() البدائيين. تدعم العربية. قابلة للتخصيص الكامل (ألوان، أحجام، أزرار). تدعم preConfirm للتحقق من المدخلات.
•	كيف يُستخدم؟ <script src="https://cdn.jsdelivr.net/npm/sweetalert2@11"></script>. core.js يغلّف دواله في RW_UI (مثل RW_UI.toast, RW_UI.showLoader, RW_UI.showConfirm).
•	الاعتماديات: لا شيء – مكتبة مستقلة.
•	من يعتمد عليها: جميع ملفات HTML (26 تطبيق + main.html).
•	تحذير: استخدام document.getElementById داخل Swal يؤدي إلى أخطاء. يجب استخدام Swal.getPopup().querySelector (المادة 31 من الدستور). أيضاً، Tailwind CSS لا يعمل داخل Shadow DOM الخاص بـ Swal، لذا يجب استخدام style مباشر.
3.1.4 Font Awesome 6
•	ماذا يفعل؟ يوفر آلاف الأيقونات الجاهزة (مثل 🏠، ⚙️، 📦، 🚚).
•	لماذا اختير؟ الأيقونات تجعل الواجهة أكثر وضوحاً واحترافية. المكتبة الأكثر شيوعاً ومجانية.
•	كيف يُستخدم؟ <link rel="stylesheet" href="https://cdnjs.cloudflare.com/ajax/libs/font-awesome/6.4.0/css/all.min.css">.
•	الاعتماديات: لا شيء.
3.1.5 Chart.js
•	ماذا يفعل؟ يوفر رسوماً بيانية (خطوط، أعمدة، دوائر) للوحة التحكم والتقارير.
•	لماذا اختير؟ لعرض KPIs واتجاهات المبيعات بصرياً في النظام الأم وتطبيقات المشرفين. خفيفة وسهلة الاستخدام.
•	كيف يُستخدم؟ <script src="https://cdn.jsdelivr.net/npm/chart.js@4.4.0/dist/chart.umd.min.js"></script>.
•	الملفات المستخدمة فيه: main.html, sales/supervisor.html, sales/manager.html, warehouse/manager.html.
3.1.6 Leaflet.js
•	ماذا يفعل؟ يوفر خرائط تفاعلية (OpenStreetMap) لعرض مواقع العملاء.
•	لماذا اختير؟ لمساعدة مندوب التوصيل في الوصول للعملاء عبر تبويب "الخريطة" في driver.html. مجاني ومفتوح المصدر (بديل لـ Google Maps المدفوع).
•	كيف يُستخدم؟ يُستدعى عبر CDN في delivery/driver.html فقط.
•	الاعتماديات: OpenStreetMap (خدمة خرائط مجانية).
3.1.7 QRCode.js
•	ماذا يفعل؟ يولد رموز QR للفواتير (للامتثال لمتطلبات ZATCA).
•	لماذا اختير؟ لعرض رمز QR على الفاتورة يمكن مسحه للتحقق من صحتها.
•	كيف يُستخدم؟ <script src="https://cdn.jsdelivr.net/npm/qrcodejs@1.0.0/qrcode.min.js"></script>.
•	الملفات المستخدمة فيه: main.html (دالة generateQRInvoiceBase64), sales/pos.html.
3.1.8 Html5Qrcode
•	ماذا يفعل؟ يوفر ماسح باركود باستخدام كاميرا الجهاز.
•	لماذا اختير؟ لتسريع عملية البحث عن الأصناف في المستودع (بدلاً من كتابة الكود يدوياً).
•	كيف يُستخدم؟ <script src="https://unpkg.com/html5-qrcode"></script>. يستخدم getUserMedia للوصول للكاميرا.
•	الملفات المستخدمة فيه: warehouse/counter.html, warehouse/returns.html.
•	المخاطر: لا تعمل على جميع المتصفحات القديمة.
3.1.9 Dexie.js (IndexedDB)
•	ماذا يفعل؟ يوفر غلافاً (Wrapper) سهل الاستخدام لقاعدة بيانات IndexedDB في المتصفح. يسمح للتطبيقات بالعمل Offline (بدون إنترنت) وتخزين البيانات محلياً.
•	لماذا اختير؟ التطبيقات الميدانية (مثل مندوب التوصيل، مندوب المبيعات) تعمل في مناطق قد لا يتوفر فيها إنترنت. Dexie.js يسمح لهم بمواصلة العمل وحفظ البيانات محلياً، ثم مزامنتها عند عودة الاتصال. هذا هو أساس استراتيجية Offline-First التي يتفوق بها الروائع.
•	كيف يُستخدم؟ <script src="https://cdn.jsdelivr.net/npm/dexie@3/dist/dexie.min.js"></script>. core.js يغلّفه في RW_DB للتوحيد. التطبيقات غير المهاجرة تعرّفه مباشرة (var db = new Dexie(...)).
•	الاعتماديات: IndexedDB API (متصفح).
•	الملفات المستخدمة فيه: تطبيقات المبيعات، تطبيقات المخازن، المتجر الإلكتروني.
•	المخاطر: ⚠️ كل تطبيق يعيد تعريف Dexie محلياً بدلاً من استخدام RW_DB الموحد من core.js. هذا دين تقني يجب سداده في P1.
•	مقارنة بالمنافسين:
o	SAP: لا يدعم Offline بشكل أصلي.
o	Odoo: يدعم Offline محدود عبر PWA.
o	الروائع يتفوق بدعم Offline First حقيقي عبر Dexie.js.

٣.٢ الخلفية (Backend – Supabase)
3.2.1 Supabase
•	ماذا يفعل؟ منصة Backend as a Service (BaaS) مفتوحة المصدر، بديل لـ Firebase. توفر قاعدة بيانات PostgreSQL، مصادقة، تخزين ملفات، Edge Functions، و Realtime.
•	لماذا اختير؟ لتسريع التطوير وتقليل تكاليف البنية التحتية. بدلاً من بناء خادم خاص، Supabase يوفر كل شيء جاهزاً. PostgreSQL كاملة (وليس NoSQL).
•	كيف يعمل؟ النظام يتصل بـ Supabase عبر @supabase/supabase-js@2. هناك نوعان من المفاتيح: ANON_KEY (للاستخدام الآمن في المتصفح) و SERVICE_ROLE_KEY (للاستخدام في Edge Functions فقط).
•	الاعتماديات: PostgreSQL، Deno، AWS.
•	من يعتمد عليها: كل تطبيق في النظام.
•	المخاطر: Vendor Lock-in. لكن Supabase مفتوح المصدر ويمكن الترحيل منه.
3.2.2 PostgreSQL
•	ماذا يفعل؟ قاعدة البيانات العلائقية. تحتوي على 51 جدولاً لجميع بيانات النظام.
•	لماذا اختير؟ أقوى قاعدة بيانات مفتوحة المصدر. تدعم العمليات الذرية (Atomic)، و RLS، و JSONB، و Stored Procedures.
•	كيف تُستخدم؟ عبر Supabase JS SDK في الواجهة، وعبر supabase (Service Role) في Edge Functions.
•	الاعتماديات: Supabase.
•	من يعتمد عليها: كل شيء.
3.2.3 Supabase Auth
•	ماذا يفعل؟ يوفر نظام مصادقة كامل (تسجيل دخول، تسجيل، إعادة تعيين كلمة مرور، JWT).
•	لماذا اختير؟ لإدارة حسابات المستخدمين بشكل آمن دون الحاجة لبناء نظام مصادقة من الصفر.
•	كيف يُستخدم؟ core.js (وحدة RW_Auth) تغلف Supabase Auth. تطبيقات PWA القديمة تستدعيه مباشرة.
•	الاعتماديات: Supabase.
3.2.4 Edge Functions (Deno/TypeScript)
•	ماذا تفعل؟ توفر منطق الخادم (Business Logic). 71 دالة تغطي كل العمليات: المبيعات، المخازن، التوصيل، المرتجعات، التسويات، التقارير.
•	لماذا وُجدت؟ لفصل منطق الأعمال عن الواجهة الأمامية. المفاتيح الحساسة (مثل SERVICE_ROLE_KEY) لا تصل للمتصفح أبداً. أيضاً، لتنفيذ العمليات الذرية (Atomic) على قاعدة البيانات.
•	كيف تعمل؟ كل دالة هي مجلد يحتوي على index.ts مكتوب بلغة TypeScript ويُشغّل على Deno. تُستدعى من تطبيقات PWA عبر fetch (أو RW_API.call). يُسمح بـ const و let في Edge Functions فقط (استثناء من المادة 1).
•	الاعتماديات: Supabase، Deno.
•	المخاطر: مجتمع Deno أصغر من Node.js. بعض المكتبات غير متوافقة.
3.2.5 Supabase Storage
•	ماذا يفعل؟ يوفر تخزيناً للملفات (مثل صور المنتجات، شعارات الشركات).
•	لماذا وُجد؟ لتخزين الصور بشكل آمن ومنفصل عن قاعدة البيانات.
•	كيف يعمل؟ core.js يستخدمه لتحميل صور المنتجات. يتم تخزينها في Buckets (مثل product-images).
3.2.6 Supabase Realtime
•	ماذا يفعل؟ يوفر تحديثات حية (WebSockets) للتطبيقات.
•	لماذا وُجد؟ لتمكين التطبيقات من استقبال التحديثات فوراً دون الحاجة لإعادة التحميل.
•	الحالة الحالية: غير مفعل بالكامل. هذه فجوة P2.
3.2.7 RLS (Row Level Security)
•	ماذا يفعل؟ يوفر أمناً على مستوى الصفوف في PostgreSQL. يضمن أن كل مستخدم لا يرى إلا البيانات المصرح له بها.
•	لماذا وُجد؟ لحماية البيانات في نظام متعدد المستأجرين (مستقبلاً) ومنع المستخدمين من الوصول لبيانات لا تخصهم.
•	الحالة الحالية: مفعل على بعض الجداول. جداول مثل driver_liabilities و daily_settlements قد لا تملك RLS. هذه فجوة أمنية.

٣.٣ النشر والاستضافة (DevOps)
3.3.1 GitHub
•	ماذا يفعل؟ مستودع الكود المصدري.
•	كيف يعمل؟ كل تعديل يُرفع إلى GitHub. Cloudflare Pages يتزامن تلقائياً مع المستودع وينشر التغييرات.
3.3.2 Cloudflare Pages
•	ماذا يفعل؟ منصة استضافة للمواقع الثابتة (Static Sites) و PWA.
•	لماذا اختير؟ استضافة مجانية (حتى حد معين)، سريعة، وتدعم النشر التلقائي من GitHub.
•	كيف يعمل؟ كل تطبيق PWA يُنشر كمشروع منفصل على Cloudflare Pages. _headers يتحكم في الكاش.
3.3.3 Service Worker (sw.js)
•	ماذا يفعل؟ يدير الكاش، يدعم العمل Offline، ويكتشف التحديثات.
•	لماذا وُجد؟ لجعل التطبيقات تعمل بدون إنترنت، وللتغلب على مشكلة الكاش التي استمرت 4 أشهر.
•	كيف يعمل؟ sw.js يُسجّل عبر register-sw.js. يجب أن يستخدم Network Only لـ HTML و API Calls (إصلاح P0).
•	الاعتماديات: register-sw.js, _headers.

٣.٤ أدوات التطوير (DevTools)
3.4.1 Node.js
•	ماذا يفعل؟ بيئة تشغيل JavaScript على الخادم.

•	كيف يُستخدم؟ node schema-validator.js.
3.4.2 schema-validator.js
•	ماذا يفعل؟ أداة فحص آلي تتحقق من عدم وجود مخالفات دستورية في الكود (const, let, invoke, innerHTML, ...).
•	لماذا وُجد؟ لمنع انهيار النظام بسبب أخطاء غير مقصودة من المساعدين الجدد. هذه الأداة لا يوجد مثيل لها في أي نظام منافس.
